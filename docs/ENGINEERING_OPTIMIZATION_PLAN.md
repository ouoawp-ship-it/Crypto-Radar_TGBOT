# 泡泡雷达工程级优化与部署目标

> 依据：《Mercu、DeepCat 与泡泡雷达三方深度分析及产品补缺方案》
>
> 执行分支：`codex/engineering-radar-optimization`
>
> 起始版本：`v1.82.0`
> 目标：把现有“自动扫描与推送系统”升级为“异常机会雷达与信号验证工作台”，不恢复研究型 Web 平台。

## 1. 产品边界

### 必须保留

- Telegram 主 Bot 的扫描、门禁、去重、冷却、话题路由和真实推送确认。
- AI Bot 的查币、解释和提醒承接。
- 启动、资金流、资金费率、结构、结构复盘与公告模块。
- SQLite 信号事实、后台服务/任务/日志/审计与 stable-check。
- 不读取交易私钥、不自动下单、不输出无证据的交易指令。

### 明确不做

- 不恢复回测、校准、模型注册、研究生命周期等已删除的研究型页面。
- 不复制 Mercu/DeepCat 的品牌、源码、文案、阈值或专有数据。
- 不让浏览器直接扇出请求多家交易所。
- 不在数据缺失时显示伪造值或把缺失值当作 `0`。
- 第一阶段不建设付费、社交、复杂账号或全网 KOL 聚合。

## 2. 北极星目标

用户收到 Telegram 信号后，应在一次点击内完成：

```text
异常被发现
  -> 规则与风险门禁
  -> Telegram 推送
  -> Web 打开同一条信号
  -> 查看数据来源、新鲜度、市场证据、同币历史
  -> 查看自身排名、市场排名、共振与生命周期
  -> 继续 AI 解释或创建提醒
```

核心衡量：

- Telegram 到信号详情深链成功率 ≥ 99%。
- 用户 10 秒内能找到并解释最重要的 3–5 个信号。
- 详情首屏的核心数据字段证据覆盖率 ≥ 80%。
- 缺失数据误显示为 `0` 的次数为 0。
- 新增接口 P95 小于 800ms；首次上游聚合可降级，缓存命中请求目标小于 200ms。
- 前端 360px、768px、1440px 三个关键宽度均可完成筛选、开详情和继续行动。

## 3. 目标架构

```text
交易所/公告数据源
  -> Python 扫描与统一字段
  -> 规则、风险门禁、信号事实
  -> SQLite SignalEventStore
  -> 服务端快照缓存与上下文聚合
  -> /public-api/* 脱敏只读接口
  -> Next.js 雷达/详情
  -> Telegram 与 AI Bot 深链

后台控制台
  -> 服务、任务、日志、审计、配置、稳定检查
```

所有公开市场字段采用统一元数据：

```json
{
  "value": 12.34,
  "unit": "percent",
  "source": "binance_futures",
  "observed_at": "2026-07-16T12:00:00Z",
  "age_sec": 8,
  "status": "fresh",
  "quality": "direct"
}
```

允许的状态：`fresh`、`stale`、`degraded`、`unavailable`。

## 6. 数据与安全门禁

- 公开接口只允许白名单字段，任何名称包含 token、secret、password、api_key 的字段必须删除或脱敏。
- 市场快照只能接受规范化 USDT 合约币种，长度和字符集受限。
- 客户端不能指定任意上游 URL、内部路径、日志文件或服务器命令。
- 所有 AI 解释必须绑定结构化证据，不得在 `unavailable` 时编造结论。
- URL、日志、分析平台不能保存密码、验证码、Bot Token、管理员 Cookie 或 API Key。
- 新增公开能力必须有单元测试；新页面必须通过 typecheck 和 production build。

## 7. 部署与回滚

每个版本执行：

1. 本地 Python 全套测试、compileall、前端 typecheck/build。
2. 创建明确提交并推送工程分支。
3. GitHub Actions 全绿后创建 PR，先审查变更边界再合并。
4. 服务器执行 `paopao update --yes` 或 `bash scripts/update_server.sh --yes`。
5. 执行 `bash scripts/check_https_deploy.sh --with-stable-check`。
6. 核对公开前台、后台、公开 API、私有 API 401、五个 systemd 服务和 Nginx。
7. 若出现阻断，使用更新脚本生成的备份/上一 Git 提交回滚；不使用破坏数据目录的方式回滚。

生产发布完成的最低标准：阻断为 0、无新增真实错误、Telegram 真实推送链路正常、公开接口不泄露敏感字段、移动端核心动作可完成。
