# v1.87.3 生产发布闭环收口

## 原因

v1.87.2 部署后的真实生产验收证明响应体积、安全头和服务状态达标，但仍暴露三个工程缺口：

- 公开情报冷请求后端中位数约 1.04 秒，超过 800ms 目标；1 天查询仍分析了 30 天读取出的全部事实。
- AI Bot Token 已配置但 `AI_BOT_USERNAME` 缺失时，stable-check 仍显示配置可用，Web 分析和提醒深链实际为空。
- Certbot dry-run 失败后验收脚本立即删除输出，却提示用户查看输出，无法定位原因。

## 修复

- 雷达情报只分析当前查询窗口；窗口外事实继续参与历史排名、共振和生命周期依据。
- 单币与单信号详情按稳定 `public_ref`/旧数字 ID 定向计算，避免为了一个详情分析全部 2,000 条事实。
- 健康门禁、稳定检查和部署验收把 AI Bot 用户名纳入完整闭环条件。
- Certbot 成功时清理临时输出；失败时打印 stdout/stderr 尾部并保留两个诊断文件。

## 语义与性能

- 2,000 条事实、约 1,200 条处于 1 天窗口的本地冷计算中位数约 233ms，最慢约 337ms。
- 单信号定向计算约 13ms。
- 与 v1.87.2 逐字段比较：当前窗口 items、四类榜单、摘要以及目标信号情报保持一致。
- 24 小时边界继续按包含处理，30 天历史仍可决定重启/增强/降温等生命周期。

## 兼容性

公开 API、SQLite schema、Telegram 推送和前端类型不变。部署后需要在配置中心填写 `AI_BOT_USERNAME`（不含 `@`）才能显示 Web→AI/提醒动作。

## 发布门禁

1. Python 全量测试、compileall 与生产规模性能保护线。
2. TypeScript、Next.js production build、npm audit、Chromium E2E。
3. GitHub Actions 全绿并合并。
4. 服务器更新后单独执行 Certbot dry-run，再运行完整 HTTPS/stable-check。
5. 外网验证冷性能、响应预算、安全头、AI/提醒深链和移动端核心动作。
