# v1.87.8 稳定版诊断精度收口

## 原因

v1.87.7 的业务与部署验收已经全部通过，但 stable-check 仍把三类历史/状态信息归为当前业务警告：

- 主服务输出的 JSON 统计字段 `"failed": 156`、`"failed": 3` 被关键词扫描误判为运行错误；系统日志过去 24 小时实际没有 `journalctl -p err` 记录。
- 最近 10 条失败 Web 审计全部是密码错误的登录尝试，不是配置保存、服务控制或任务执行失败。
- 两条已删除研究型平台遗留的 `lifecycle-outcome-reconcile` 历史任务仍进入当前任务健康度。

## 修复

- 纯 JSON 的 `error/errors/failed/failures` 数值或空值字段不再计入日志错误，也不伪装成网络重试噪声。
- Web 审计新增 `all`、`operations`、`auth` 范围；稳定版健康度只读取业务操作失败，登录失败原始记录继续完整保留并单独暴露计数。
- 后台任务统计支持按当前任务类型过滤；运维快照只统计现有六类任务，已下线任务仍保留在 SQLite 中供历史审计。
- 部署报告不再根据 `0.0.0.0:8080` 推断公网可访问，明确提示公网暴露由防火墙和云安全组决定。

## 安全与数据影响

不会删除或改写登录审计、任务历史和服务日志；只修正当前健康度的分类口径。公网实测 `52.198.75.229:3000` 与 `:8080` 均连接超时，Nginx 后端端口未直接暴露。

## 回归保护

- 验证 JSON 失败计数与真实 `RuntimeError` 能正确区分。
- 验证登录失败和业务操作失败分别查询。
- 验证已下线任务可从当前统计排除，同时全量历史仍可读取。
