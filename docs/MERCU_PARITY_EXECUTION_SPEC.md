# Mercu Radar / Info / Funds 等价实现与验收契约

更新日期：2026-07-18

## 1. 项目边界

本阶段只重做一个独立的 Web 市场工作站模块：Radar、Info、Funds。原有 Telegram BOT 的五个推送模块保持独立，不把 BOT 筛选、推送状态或运维控件混入 Mercu 等价页面。

不复刻目标站的个人空间；不复刻目标站的 AI 智选产品。导航保留 Paoxx 自有的 AI 智选占位，后续使用 Paoxx 自有模型、提示词和证据链实现。

目标站的登录态用于观察公开呈现的界面、交互和数据语义。实现不得读取、保存或转发目标站的 Cookie、Token、用户数据和私有接口凭据；不得声称拿到了目标站未公开的服务端源码或精确公式。

## 2. “1:1”完成定义

只有同时满足以下四层，才允许在发布说明中写“Radar / Info / Funds 复刻完成”：

1. 视觉层：导航、栅格、面板、间距、字号、颜色、边框、滚动容器、空态和加载态与目标基线一致。
2. 交互层：时间窗、榜单模式、搜索、展开、悬浮说明、暂停/刷新、详情跳转和跨模块联动一致。
3. 数据语义层：同名指标表达相同含义；金额榜、强度榜、历史排名、共振、加速度和信息分栏不能用近似但不同的指标冒充。
4. 工程层：数据来自 Paoxx 自有或明确授权的数据源，公式可解释、可测试、可回放，降级时不伪造数据。

目标站私有服务端公式无法从可见页面证明时，采用“行为与语义等价的独立实现”，并在方法说明中公开 Paoxx 口径。不得把推断包装成目标站原公式。

## 3. 已观察到的目标基线

### 3.1 全局外壳

- 登录态真实路由分别为 `/?tab=radar`、`/?tab=info`、`/?tab=fund`；Funds 使用单数 `fund`，`/?tab=funds` 会回落到其他页面，禁止用作资金页截图或 DOM 基线。
- 顶栏固定高度 56px，包含 Radar、Info、Funds、AI、实时状态、时钟、主题、收藏、语言和账户入口。
- Radar 主区在 1440×900 基线下为三列：左列约 268px，中列约 864px，右列约 268px；列间距 10–12px。
- 页面采用固定视口高度和面板内部滚动，不是普通长页面。
- 主暗色令牌：`#08090a`、`#0d0e10`、`#141517`、`#1a1c1e`、`#1d1f22`；正文 `#e9eaec`，弱文本 `#abb1b9`，上涨 `#3fb27f`，下跌 `#e0555c`，警告 `#cda24a`，强调 `#cdb45f`。
- 字体体系为 DM Sans / Inter、JetBrains Mono、Noto Sans SC；数字和排名使用等宽字体。

### 3.2 Radar

目标页面可观察的主要区域与行为：

- 异动监控：OI 暴涨/暴跌、成交量爆发、价格异动、现货/合约主力流入流出和状态升级。
- 每条异动同时展示自身历史强度排名、全场强度排名、全场绝对量级排名。
- 主力监控：聚焦 BTC/ETH 大额订单流。
- 基差·费率监控：跨币种、跨交易所的基差和资金费率极端值。
- 热钱观察榜：15m、30m、1h、4h、1d 五个时间窗。
- 每个 Momentum 窗口包含涨幅、跌幅、持仓、主力合约流入/流出、主力现货流入/流出，并区分量级榜与强度榜。
- 共振使用五个时间窗的上榜位图表达；位图顺序必须固定。
- 飙升榜：1h 滚动、加速度排序、Top 5。
- 异动总榜：24h 累积异动次数、Top 14，并带算法标签。
- 埋伏池：持仓累积、价格相对安静、尚未明显启动的标的。
- AI 多头/空头关注属于目标业务的衍生展示；Paoxx 第一阶段只保留可解释的规则关注，不冒充目标 AI。

目标前端可观察到的接口分工：

- `radar/anomaly-v4`：异动流。
- `radar/momentum`：分别请求 15m、30m、1h、4h、1d。
- `radar/rank`：累计/综合榜。
- `radar/surge`：加速度榜。
- `info/radar-briefs`：雷达摘要与信息模块联动。

Paoxx 不直接调用这些目标接口；只参照其分工设计自己的公开契约。

### 3.3 Info

目标页面是固定高度的四列信息工作站，不是卡片分页列表：

- 顶部 AI 信息蒸馏条，可展开 4h 综合摘要；第一阶段改成 Paoxx 规则蒸馏，AI 版本后置。
- 聚合资讯列。
- 英文流资讯列。
- KOL 聚合资讯列。
- 币安广场情绪列。
- 信息按 8h 主线、4h 重点、1h 近况、30m 突发分层。
- 包含搜索、来源、时间、关联币种、情绪/重要度、原文跳转、加载态和空态。
- 右侧/辅助榜包含 4h 活力榜与 24h 情绪总榜。

目标前端可观察到的数据分工包括 dashboard、briefs、brief、plaza、deep、radar-briefs，以及中文资讯、英文流和 KOL 三路 feed。

当前 Paoxx Info 只有官方公告聚类与分页卡片，因此不满足结构或数据覆盖要求。

### 3.4 Funds

目标导航语义为资金流工作站。已确认目标单币资金视图至少包含：

- 合约/现货切换和 K 线。
- 16h、2d、4d、5d、15d、60d 资金跨度。
- 合约资金流、现货资金流、累计净流入/流出与价格叠加。
- 跨交易所 OI 对比、OI 历史、资金费率历史。
- 主力资金、当前币价、命中率和持续时间。
- Top50 集中度趋势、交易所净流、链上、鲸鱼活动。
- 价格表现、关键价位、POC、VAH、VAL 和相关新闻。

Funds 根页面的最终列宽、全部控件和状态仍须以登录态截图逐项固化；完成该基线前不得声称 Funds 已 1:1。

当前 Paoxx Funds 的板块气泡图和资产表可以作为数据探索页保留，但不能作为 Mercu Funds 等价页。

## 4. Paoxx 独立计算口径

以下口径必须在接口返回的 `methodology` 中公开，并由测试固定：

### 4.1 自身历史强度

对同一币种使用封闭的 5m 基础桶，分别聚合 5m、15m、1h 指标。当前值与最近 24h 同口径样本比较：

`self_rank = 1 + count(history_strength > current_strength)`

`self_percentile = count(history_strength <= current_strength) / valid_samples * 100`

样本不足时返回 unavailable，不补假排名。

### 4.2 全场强度与全场量级

- 全场强度：对不同币种的相对异常得分做同一时点横截面排序，避免大市值币天然占优。
- 全场量级：对绝对美元金额、绝对 OI 变化额或绝对成交额做横截面排序，保留大资金规模语义。
- 两种榜单不得复用同一数值或只改标题。

### 4.3 Momentum 与共振

- 窗口固定为 15m、30m、1h、4h、1d。
- 每个窗口使用已闭合数据聚合价格变化、OI 变化、现货净流、合约净流。
- 量级榜按绝对数值排序；强度榜按相对自身历史基线的经验分位或稳健标准分排序。
- 共振位图仅表示该币是否进入对应窗口的指定榜单；方向共振必须另行计算，不能从“上榜”直接推断多空。

### 4.4 Surge

当前闭合窗口与前一等长窗口比较：

- 成交加速度、净流加速度、OI 加速度分别标准化。
- 价格方向只决定多/空标签，不替代资金加速度。
- 总分、触发阈值、覆盖率和样本数必须随接口返回。

### 4.5 埋伏池

至少同时满足：

- 15m OI 或净流为正向累积。
- 同期价格绝对变化低于安静阈值。
- 当前未触发 Surge。
- 数据覆盖和新鲜度达标。

### 4.6 Funds

- 现货/合约净流基于主动买入成交额减主动卖出成交额（CVD 口径）。
- 累计值按闭合桶累加；不得把成交额直接标成净流入。
- OI 变化同时返回百分比和美元变化额。
- 费率返回单周期值、周期长度和可比年化值；年化必须注明只是近似外推。
- 跨所数据按统一合约面值和美元计价归一化。

## 5. Paoxx API 分层目标

前端不得再用一个聚合接口伪装目标的全部行为。Paoxx 自有接口至少拆分为：

- `/public-api/workstation/radar/anomalies`
- `/public-api/workstation/radar/momentum?window=15m|30m|1h|4h|1d`
- `/public-api/workstation/radar/rank`
- `/public-api/workstation/radar/surge`
- `/public-api/workstation/radar/briefs`
- `/public-api/workstation/info/dashboard`
- `/public-api/workstation/info/feed?channel=news|en|kol|plaza`
- `/public-api/workstation/info/briefs`
- `/public-api/workstation/funds/overview`
- `/public-api/workstation/funds/assets`
- `/public-api/workstation/funds/series?symbol=...&kind=spot_flow|futures_flow|oi|funding`

每个接口必须返回 schema_version、generated_at、data_status、coverage、warnings 和 methodology。接口可以复用现有采集与存储，但展示契约必须独立于 BOT 推送模型。

## 6. 实施顺序

1. 冻结旧页面为对照，不继续在旧结构上追加面板。
2. 完成三个页面的登录态截图、DOM、控件和空/错/加载状态矩阵。
3. 建立 workstation API 和单元测试，优先复用现有封闭分钟、排名、Surge、Funds 与新闻聚类能力。
4. 重做 Radar，完成五窗口与三类排名。
5. 重做 Info，完成四列信息流与分层。
6. 重做 Funds，完成总览、资产与单币资金时序。
7. 执行视觉差分、交互回归、数据回放、性能和移动端验收。
8. Web 验收通过后，再把可靠指标通过适配层输入原有 BOT 五模块。
9. 最后提交、推送、CI、部署和线上验收。

## 7. 发布门禁

- 1440×900 与 1920×1080 主基线的结构尺寸误差不超过 1px；实时文本和图表数据使用固定 fixture 后比较。
- 颜色、字号、字重、间距、边框和滚动行为逐项快照测试。
- 所有交互有 Playwright 回归测试；空数据、慢请求、局部接口失败和全局失败均有状态覆盖。
- 数据回放测试证明同一输入产生稳定排名、共振、Surge 和累计资金流结果。
- 页面不得调用 Mercu 域名或其后端域名。
- 页面不得显示 BOT 运维状态、推送开关或 BOT 专用筛选器。
- 构建、类型检查、Python 单测、Web E2E、性能预算和线上 stable-check 全绿。

在以上门禁全部通过前，对外状态统一写“实现中”，不能写“已 1:1 完成”。
