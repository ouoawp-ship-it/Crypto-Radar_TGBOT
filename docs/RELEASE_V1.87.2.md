# v1.87.2 信号情报冷计算加速

## 原因

v1.87.1 解决了公开响应体积和前台安全头问题，但生产规模压测发现：2,000 条事实的首次情报计算中位数约 1.11 秒，高于工程计划中的 800ms 发布目标。剖析显示，每条信号会重复扫描全部事件、重新排序同模块候选，并反复解析相同绝对规模字段。

## 修复

- 按模块、币种、币种+模块建立升序时间索引。
- 使用二分边界提取 30 天自身历史、市场窗口和五档共振窗口。
- 同一事实的成交额、OI 或市值只解析一次。
- 保留同时间戳包含、按 `ts/id` 选取每币最新、排名样本和生命周期依据的原有语义。
- 增加 2,000 条生产规模的 1 秒冷计算回归保护线。

## 本地基线

Windows 本地 2,000 条、180 个币种、5 个模块、7 次冷计算：

- 优化前中位数约 1.11 秒；
- 优化后中位数约 340ms，最慢约 471ms；
- 300 条混合顺序事实在 1h、1d、30d 三个窗口下与旧算法逐字段一致。

## 兼容性

公开 API、SQLite schema、Telegram 推送、AI Bot、排名口径和前端类型均不变，无需数据迁移。

## 发布门禁

1. 2,000 条性能测试通过。
2. Python 全量测试和 compileall 通过。
3. TypeScript、production build、npm audit、Chromium E2E 通过。
4. GitHub Actions 全绿后合并。
5. 服务器更新后执行完整 HTTPS/stable-check 与外网性能验收。
