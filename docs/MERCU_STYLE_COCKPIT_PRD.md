# 泡泡雷达 Mercu 风格情报驾驶舱 V2 产品需求文档

> 文档版本：v1.0
> 编写日期：2026-07-16
> 对应项目：泡泡抓币 Crypto Radar v1.88.0
> 参考产品：Mercu 登录后 Web 产品
> 文档用途：产品范围确认、交互设计、数据契约、工程拆分、测试与上线验收

## 1. 执行结论

本项目可以实现与 Mercu 相同类型的完整产品体验，包括实时异动雷达、热钱榜单、板块资金、全市场资产表、信息聚合、AI 决策驾驶舱、自选与单币详情闭环。

实现目标是“能力与体验同等级”，不是复制对方源码、品牌、专有文案、内部算法阈值或受限制的数据。Mercu 的后端源码、私有数据源、评分权重和 AI 提示词无法通过页面确认，因此泡泡雷达必须基于自己的数据、可验证规则和品牌视觉独立实现。

泡泡雷达并非从零开始。当前项目已经具备以下可复用基础：

- Python 雷达主服务、资金费率、资金流、启动与公告监控。
- SQLite 信号库、公开信号 API、运行时缓存、限流与数据状态字段。
- 自身历史排名、市场强度排名、市场绝对规模排名。
- 15m、30m、1h、4h、1d 共振窗口与信号生命周期。
- Next.js 信号雷达、详情抽屉、单币上下文、自选列表与移动端 E2E。
- Telegram 推送、信号深链、AI Bot、价格提醒和后台运维闭环。
- CI、部署脚本、HTTPS 验收、服务健康检查、日志与审计。

核心差距不是“有没有信号”，而是现有后端能力尚未被组织成连续、可横向比较、可快速验证的市场驾驶舱。V2 应重点补建连续市场快照、统一榜单、板块资金、信息聚合和结构化 AI 解盘。

## 2. 产品定位

### 2.1 一句话定位

面向加密市场观察者的实时异动情报驾驶舱：自动发现异常，解释异常为什么重要，并通过 Web、Telegram 与 AI 助手形成持续跟踪闭环。

### 2.2 产品原则

1. 先给事实，再给解释：价格、成交量、OI、现货与合约资金、资金费率等结构化数据优先。
2. 先扫面，再聚焦：首屏帮助用户看清全市场，再进入单币验证。
3. 相对强度与绝对规模并存：避免只看百分比或只看大币金额。
4. AI 只做证据压缩：AI 不负责生成原始市场数字，也不得在数据缺失时补写结论。
5. Web 与 Bot 使用同一事实源：同一信号在网页、Telegram 和 AI Bot 中必须语义一致。
6. 不自动交易：系统只做监控、解释、风险提示、提醒和深链，不保存交易私钥、不自动下单。
7. 真实状态优先：加载、过期、降级、不可用必须明确展示，不能用旧数据冒充实时数据。

### 2.3 非目标

- 不恢复已删除的研究型平台、通用回测、校准、模型注册和结构突破/结构复盘模块；实时异常允许提供有样本门槛的轻量离线方向结果统计。
- 不建设交易所账户绑定、自动下单、跟单或资产托管。
- 不照搬 Mercu 品牌名称、Logo、专有文案和视觉资产。
- 不在 P0/P1 阶段建设付费订阅、邀请体系、复杂社交关系或 KOL 账号运营后台。
- 不将来源不明的大额交易直接描述为“机构”或“聪明钱”。

## 3. 目标用户

### 3.1 高频市场观察者

需要在同一屏幕内快速判断全市场正在发生什么，重点关注价格、成交量、OI、资金流和费率异常，不希望逐币切换交易所页面。

### 3.2 Telegram 提醒用户

平时不持续打开网页，依赖 Bot 被动接收高价值信号；收到推送后需要一键打开对应信号与单币上下文，并继续设置提醒或向 AI 追问。

### 3.3 风险与机会验证者

已经看到某个币种异动，希望确认它在自身历史中是否罕见、全市场排名如何、多个周期是否共振、现货和合约是否同向，以及信号正处于哪个生命周期。

### 3.4 系统管理员

负责数据源、规则阈值、服务运行、Telegram 路由、AI 配置、失败任务、日志和部署稳定性，不参与公开前台日常使用。

## 4. 核心使用场景

### 4.1 全市场扫面

用户打开雷达，十秒内回答：

- 当前市场整体偏流入、流出还是分歧？
- 哪些币正在发生新的高强度异常？
- 哪些币同时进入涨幅、OI、现货资金或合约资金榜？
- 哪些事件只是小币百分比异常，哪些具有足够大的绝对量级？

### 4.2 从异常进入验证

用户点击异动事件后，在抽屉中完成：

1. 查看事件方向、时间窗和原始指标。
2. 查看自身历史、市场强度、市场绝对量级三类排名。
3. 查看 15m 至 1d 共振和 NEW/增强/持续/降温/重启/失效状态。
4. 查看 K 线、资金流、OI、费率、相关新闻和同币历史事件。
5. 加入自选、设置 Telegram 提醒或向 AI Bot 追问。

### 4.3 资金与板块轮动

用户在资金页查看整体净流入、板块气泡、流入/流出龙头和资产明细表，从板块层快速定位到币种层。

### 4.4 信息与行情交叉验证

用户在信息页同时查看官方公告、中文资讯、英文资讯和市场情绪。信息必须能关联到币种，并可反向打开该币行情与最近异常。

### 4.5 AI 决策压缩

用户不想逐项阅读指标时，通过全局 Agent、BTC/ETH 解盘 Agent、异常候选 Agent 和消息 Agent 获取结构化结论，并随时展开证据来源。

### 4.6 Web 与 Telegram 闭环

后台发现信号后写入统一事件库，Telegram 推送携带公开引用；用户点击深链进入 Web 详情，再加入自选、创建提醒或向 AI 提问。Web 中的动作回到 Bot 后仍携带 symbol、signal_ref 和时间窗上下文。

## 6. 全局应用框架

### 6.1 桌面端结构

- 顶部固定导航，高度 56 至 64px。
- 左侧品牌与主要导航：总览、雷达、资金、信息、AI 决策。
- 右侧：LIVE 状态、数据更新时间、明暗主题、自选、语言预留、用户入口预留。
- 驾驶舱页面使用全宽布局，不继续占用 256px 固定侧栏。
- 普通详情与管理页可使用最大宽度 1440px 的内容容器。
- 所有行情数字使用等宽数字特性 `tabular-nums`。

### 6.2 移动端结构

- 顶部只保留品牌、数据状态与必要操作。
- 主要导航改为底部导航或横向可滚动标签，不显示桌面侧栏。
- 雷达按“全场态势 → 异动流 → 重点榜单”单列排列。
- 详情抽屉改为全屏 Sheet，支持返回键、下滑关闭和固定底部操作栏。
- 不允许通过缩放桌面三栏来适配移动端，不允许页面产生整体横向滚动。

## 7. 页面详细需求

## 7.1 信号雷达 `/radar`

### 7.1.1 页面目标

在一个桌面视口内完成“看全局、找异常、比强弱、进详情”。这是 V2 的核心页面。

### 7.1.2 桌面布局

在宽度不小于 1440px 时采用三栏：

- 左栏 280px：实时异动监控。
- 中栏 `minmax(640px, 1fr)`：热钱观察榜单。
- 右栏 300px：资金倾向性与全场态势。
- 栏间距 10 至 12px，页面内边距 10 至 16px。
- 三栏独立滚动，顶部筛选与状态栏可吸顶。

在 1024 至 1439px 时采用两栏：左侧异动流约 320px，右侧主榜单；全场态势移到主榜单上方。低于 1024px 时改为单列。

### 7.1.3 异动监控区

顶部功能：

- LIVE/暂停更新状态。
- 扫描资产数量与最近更新时间。
- 币种搜索。
- 事件类型筛选：价格、成交量、OI、现货资金、合约资金、费率、公告、启动。
- 时间窗筛选：5m、15m、30m、1h、4h。
- 只看自选与只看高置信度。

每条事件最少包含：

- 时间、币种、事件类型和方向。
- 变化值、百分比、绝对金额或 OI 金额。
- 一条可读摘要，不超过两行。
- 自身历史极端度、市场相对强度、市场绝对量级。
- 共振周期数量与生命周期状态。
- 数据新鲜度与来源提示。

排序默认按“新鲜度 × 强度 × 数据质量”，允许切换最新、最强和最大量级。

### 7.1.4 热钱观察榜单

顶部周期：15m、30m、1h、4h、1d。

首版提供四组榜单：

1. 涨幅榜 / 跌幅榜。
2. OI 增长榜 / OI 下降榜。
3. 合约净流入榜 / 合约净流出榜。
4. 现货净流入榜 / 现货净流出榜。

每组支持“量级榜”和“强度榜”：

- 量级榜按统一美元口径的绝对值排序。
- 强度榜按相对自身基线标准化后的分位数排序。
- 每行显示排名、币种、核心值、变化率、五周期命中格、最新状态。
- 同一币进入多个榜单时显示“2榜/3榜/4榜”合流标签。
- 点击行打开单币详情；悬停显示完整方法与更新时间。

### 7.1.5 资金倾向性

展示最近周期内资产的方向合流：

- 现货净流方向。
- 合约净流方向。
- OI 方向。
- 价格方向。
- 合流数量和分歧标签。

禁止将合流直接解释为做多/做空指令。文案使用“流入倾向、流出倾向、方向分歧、证据不足”。

### 7.1.6 全场态势

展示：

- 全市场现货净流入/流出。
- 全市场合约净流入/流出。
- 总 OI 变化。
- 上涨/下跌资产数量和市场广度。
- 本周期相对上一周期的变化。
- 数据覆盖率、缺失交易所和最后成功更新时间。

### 7.1.7 筛选区交互

- 主筛选使用紧凑分段按钮，不用多行大表单。
- 次要筛选放入可展开面板。
- 已选筛选转成可删除 Chip，并提供“一键清除”。
- URL 同步 `symbol`、`type`、`window`、`sort`、`watchlist`，支持刷新与分享。
- 自动更新不能重置用户滚动、筛选和已打开抽屉。
- 新事件到达时只显示“新增 N 条”提示；用户确认后插入，避免列表跳动。

## 7.2 资金中心 `/funds`

### 7.2.1 页面结构

- 顶部切换：现货、合约；周期 15m、30m、1h、4h、1d。
- 左侧约 40%：板块资金概览与气泡图。
- 右侧约 60%：全市场资产资金表。
- 移动端先显示摘要卡和流入/流出 Top 5，完整表格改为字段可切换的卡片列表。

### 7.2.2 板块资金

首屏指标：总净流入、流入总额、流出总额、覆盖资产数、流入龙头板块、流出龙头板块。

板块气泡：

- 气泡面积代表绝对净流金额。
- 颜色代表流入、流出或中性。
- 透明度或描边代表数据完整度。
- 点击板块联动筛选右侧资产表。
- 小屏幕提供排序列表作为等价替代，不能只依赖气泡图。

建议初始分类：L1、L2、DeFi、Meme、AI、RWA、DePIN、GameFi、BTC 生态、支付、隐私、预言机、跨链、质押、交易平台币、稳定币和其他。分类表由项目维护并记录版本。

### 7.2.3 资产资金表

字段：

- 自选、币种、价格、24h 涨跌。
- 净流入、净流入变化。
- 成交额、成交额变化。
- 流入、流出。
- OI、OI 变化、资金费率。
- 市值、板块、更新时间、数据状态。

功能：搜索、多列排序、板块筛选、数据状态筛选、列显隐、分页或虚拟滚动、复制币种、加入自选、打开详情。

## 7.3 信息中心 `/info`

### 7.3.1 页面目标

把“发生了什么消息”和“市场是否已经反应”放在同一工作台中，而不是建设普通新闻门户。

### 7.3.2 桌面布局

四列或两列可配置信息流：

1. 官方公告：优先复用现有 Binance 公告监控。
2. 中文资讯：仅接入有授权或明确可聚合的数据源。
3. 英文资讯：保留原文链接与来源。
4. 市场情绪：讨论热度、正负倾向、4h 活力与 24h 趋势。

每条信息包含时间、来源、标题、摘要、关联币种、重要级别、原文链接、AI 解读状态和版权来源标记。

### 7.3.3 AI 解读

- 只对高重要度、已去重的信息生成摘要。
- 输出“事实摘要、可能影响、关联资产、需验证项”。
- 明确区分原文事实与模型推断。
- 数据或原文不可访问时不生成伪摘要。
- 同一新闻多来源合并为事件簇，保留全部来源链接。

### 7.3.4 P2 边界

KOL 聚合必须使用官方 API、授权数据或用户自行添加的公开源；不得通过绕过登录、反爬或复制全文实现。P2 可先只做官方公告和已授权资讯。

## 7.4 AI 决策 `/agents`

### 7.4.1 全局 Agent

展示最近 4h 的市场阶段、情绪指数、上涨广度、现货净流、合约净流和 OI 变化。结论由规则引擎先生成结构化状态，再由 AI 压缩成两至四句。

### 7.4.2 BTC/ETH 解盘 Agent

BTC 与 ETH 并排展示：

- 当前价格与 1h、4h、1d 方向。
- OI、资金费率、现货/合约净流。
- 拥挤度、分歧与生命周期。
- 方向判断、置信度和反证条件。

### 7.4.3 异常候选 Agent

分为偏强观察、偏弱观察和高风险观察，不使用直接下单语言。每个候选必须附带：

- 触发事件。
- 核心指标证据。
- 三类排名与多周期共振。
- 数据质量。
- 反向证据或风险。
- 打开详情、加入自选、Telegram 追问。

### 7.4.4 消息 Agent

每 15 分钟扫描新增的重要公告和资讯，只输出新增或结论发生变化的事件。每条结论可追溯到原文和关联行情。

### 7.4.5 AI 安全门禁

- 输入只来自带时间戳、来源和状态的结构化事实。
- 关键数值由代码格式化后注入，不允许模型计算原始值。
- `unavailable` 与 `degraded` 数据不得用于肯定性结论。
- 生成结果保存证据引用、模型、版本、生成时间和过期时间。
- 所有结论标记“市场观察，不构成投资建议”。

## 7.5 自选 `/watchlist`

### 7.5.1 有数据状态

每个资产显示价格、24h 涨跌、最近异常、当前生命周期、现货/合约/OI 方向、数据新鲜度和提醒状态。

操作：排序、删除、自选分组、打开详情、只看该币信号、创建提醒。

### 7.5.2 空状态

- 标题：“还没有关注的资产”。
- 说明：“在雷达、榜单或详情中点击星标，即可在这里持续跟踪。”
- 主操作：“去信号雷达”。
- 次操作：“查看资金榜单”。
- 可展示 3 个基于当前热度的推荐资产，但必须标记推荐依据。

### 7.5.3 存储演进

P0 延续浏览器本地自选；引入用户账户后迁移为服务端存储，并提供本地合并提示。匿名用户不应被强制登录。

## 7.6 单币详情 `/coin/[symbol]` 与详情抽屉

### 7.6.1 顶部摘要

币种、价格、24h 涨跌、市值、总 OI、24h 成交额、自选、数据状态、更新时间。

### 7.6.2 主要区域

- K 线：1m、5m、15m、1h、4h、1d；现货/合约切换。
- 事件时间线：按日期分组，支持类型和周期筛选。
- 现货资金与合约资金曲线。
- OI 与资金费率，多交易所对比。
- 三类排名、共振与生命周期。
- 相关公告/资讯。
- 同币历史信号与最近状态变化。

### 7.6.3 详情动作

- 加入/移出自选。
- 只看该币雷达事件。
- 创建价格、涨跌幅、OI、资金费率或 AI 条件提醒。
- 打开 Telegram AI 追问，并携带 symbol 与 signal_ref。
- 复制可分享链接。

### 7.6.4 图表策略

## 10. 排名与聚合逻辑

### 10.1 自身历史极端度

同一币种、同一指标、同一周期与同一方向下，对滚动历史样本计算经验分位数。建议至少 30 个有效样本；不足时显示“样本不足”，不输出伪精确排名。

### 10.2 市场相对强度

先把每个币当前变化相对自身历史基线标准化，再在同一时刻的全市场截面排序。可以使用稳健 Z 分数、MAD 或经验分位数，算法版本必须写入结果。

### 10.3 市场绝对量级

按统一美元口径对成交额、OI 或资金流绝对值排序。不同指标不得混排，缺少美元换算或价格来源时标记不可用。

### 10.4 多周期共振

共振必须说明“什么指标在哪些周期同向”，不能只统计信号条数。建议区分：

- 同指标多周期共振。
- 多指标同方向共振。
- 现货与合约合流。
- 价格与 OI 分歧。

### 10.5 生命周期

延续现有 NEW、增强、持续、降温、重启、失效；升级依据从单一规则分变化扩展为强度、排名、共振和持续时间，并保留 `basis` 文本。

### 10.6 板块资金

先按资产映射聚合原始资金，再计算板块净流。板块内覆盖率低于阈值时显示“覆盖不足”。稳定币、重复映射、跨板块资产和极端值需要明确处理规则。

## 11. API 契约规划

保留现有 `/public-api` 命名和统一响应信封，新增接口必须包含 `schema_version`、`generated_at`、`data_status`、`coverage` 与可选 `warnings`。

| 接口 | 作用 | 缓存建议 |
| --- | --- | --- |
| `GET /public-api/market/overview` | 全场态势与市场广度 | 10 至 15 秒 |
| `GET /public-api/radar/events` | 异动事件流、游标和筛选 | 5 至 10 秒 |
| `GET /public-api/radar/boards` | 八类榜单与合流标记 | 15 至 30 秒 |
| `GET /public-api/funds/sectors` | 板块资金聚合 | 30 至 60 秒 |
| `GET /public-api/funds/assets` | 全市场资产资金表 | 15 至 30 秒 |
| `GET /public-api/coin/context` | 扩展单币完整上下文 | 15 至 30 秒 |
| `GET /public-api/info/feed` | 公告和授权资讯 | 30 至 120 秒 |
| `GET /public-api/agents/overview` | 结构化 Agent 结论 | 60 至 300 秒 |
| `GET /public-api/watchlist/market` | 自选批量状态 | 15 至 30 秒 |
| `GET /public-api/stream` | SSE 增量事件和状态 | 心跳 15 秒 |

兼容策略：

- 现有 `/signals`、`/signals/context`、`/radar/intelligence`、`/market/snapshot` 和 `/market/watchlist` 继续可用。
- 新页面优先使用新增聚合接口，旧页面在灰度期间不受影响。
- 大响应支持字段投影、分页和压缩；禁止首屏下载完整历史。
- SSE 只传事件引用和轻量摘要，详情按需请求。

## 12. 数据源与采集策略

### 12.1 P0 可直接复用

- 当前多交易所资金费率数据源。
- 当前 Binance 行情、OI、成交额与公告。
- 当前资金流、启动和雷达规则输出。
- 当前信号 SQLite、运行时缓存和 Telegram 路由。

### 12.2 必须补建

- 统一的 1m/5m/15m/30m/1h/4h/1d 市场快照存储。
- 现货与合约资金流的统一口径和来源覆盖说明。
- 资产主数据与板块分类版本。
- 榜单预计算与历史基线。
- 信息事件去重、关联币种和版权状态。
- Agent 证据引用与结果缓存。

### 12.3 数据口径边界

“净流入”必须注明是主动买卖成交差、交易所流入流出还是自定义订单流估算，三者不得混称。没有可验证账户身份时不使用“机构净流入”。

## 13. Web、Bot 与后台协作关系

```text
交易所/公告/授权资讯
        ↓
采集与标准化 → 连续市场快照 → 规则/排名/榜单 → 统一事件与证据服务
                                               ↓
                         ┌─────────────────────┼─────────────────────┐
                         ↓                     ↓                     ↓
                    Next.js Web           Telegram Bot          AI Assistant
                    扫面与验证              主动推送              解释与提醒
                         └─────────────────────┼─────────────────────┘
                                               ↓
                                      自选、提醒、审计与运维后台
```

- Web、Telegram 和 AI Bot 不各自计算市场事实，只读取统一证据服务。
- Telegram 消息携带 `public_ref` 和 `symbol`，Web 深链打开相同事件。
- Web 创建提醒后交给现有提醒服务和 AI Bot 执行。
- 后台配置阈值、数据源和路由，但不能直接修改历史事实。
- 每次推送保存规则版本、证据快照和目标路由，方便审计。

## 14. 交互状态规范

### 14.1 加载状态

- 首次加载使用与最终布局一致的 Skeleton，禁止整页转圈。
- 各栏独立加载；资金页失败不应阻塞异动流。
- 抽屉先显示事件摘要，再渐进加载图表和历史。
- 超过 2 秒显示“正在连接数据源”；超过超时显示重试与最后可用数据。

### 14.2 空状态

区分三类：

1. 真实空：当前周期没有满足条件的事件。
2. 筛选空：当前条件无匹配，提供清除筛选。
3. 数据空：数据源不可用或覆盖不足，提供诊断信息。

禁止用“暂无数据”覆盖所有情况。

### 14.3 错误与降级

- `fresh`：正常实时数据。
- `stale`：超过目标新鲜度但仍可参考，显示具体时间。
- `degraded`：部分来源失败，显示覆盖率与缺失来源。
- `unavailable`：不展示结论，允许重试。
- 全局顶部只显示聚合状态，面板内部显示局部状态。

### 14.4 实时更新

- 连接状态：LIVE、重连中、已暂停、离线。
- 页面隐藏时降低刷新频率；恢复时请求增量。
- 断线重连采用退避和抖动，避免同时击穿后端。
- 用户滚动阅读时不自动插入导致位置跳动。

### 14.5 反馈与可访问性

- 所有图标按钮有可读名称和键盘焦点。
- 正负方向不只依靠颜色，同时使用箭头、文字和符号。
- 表格支持键盘浏览，抽屉有焦点锁定与 Esc 关闭。
- 动效遵循 `prefers-reduced-motion`。

## 15. 移动端适配验收

必须覆盖 360、390、768、1024、1440 和 1920px。

在 390px 下：

- 页面整体无横向滚动。
- 顶部主要状态一屏可见。
- 筛选可折叠，已选条件可快速清除。
- 异动卡核心信息不超过三层，次要证据展开查看。
- 榜单默认 Top 5，查看更多进入独立列表。
- 详情为全屏 Sheet，底部操作始终可达。
- 表格转换为卡片或仅显示用户选择的 3 至 4 个关键字段。

Mercu 当前移动端仍保留桌面三栏并产生压缩/横向溢出；泡泡雷达不得复制这一缺陷。

## 16. 非功能要求

### 16.1 性能

- 缓存命中时公开聚合 API P95 小于 600ms。
- 公开首屏 LCP 小于 2.5s，INP 小于 200ms，CLS 小于 0.1。
- 首屏只获取可见区域和 Top N；完整表格按页加载。
- 公开情报接口保持响应预算，默认压缩后不超过 250KB。
- 列表超过 200 行使用虚拟滚动。

### 16.2 稳定性

- 单一交易所故障不得使全站不可用。
- 聚合任务可重试、可幂等、可审计。
- 榜单和 Agent 结果带版本、时间和来源覆盖。
- 保留现有稳定检查、部署验收和 systemd 服务监控。

### 16.3 安全

- 公开接口继续限流与字段白名单。
- 管理接口保持鉴权、锁定、审计与安全响应头。
- 前端不暴露交易所密钥、Bot Token、AI Key 或内部阈值。
- 外部资讯内容做 HTML 清洗，链接使用安全属性。
- 用户凭据和第三方登录会话不得写入代码、日志、PRD 或测试夹具。

### 16.4 可观测性

至少记录：采集成功率、数据延迟、覆盖资产数、榜单生成耗时、SSE 连接数、公开 API P95、前端错误、AI 失败率、推送成功率与深链打开率。

## 17. 工程架构建议

### 17.1 后端分层

1. `collectors`：交易所、公告和资讯采集。
2. `normalizers`：统一 symbol、时间窗、单位、方向和来源。
3. `market_store`：连续快照和聚合结果。
4. `intelligence`：异常、排名、共振、生命周期和榜单。
5. `public_api`：稳定的只读契约、缓存、限流和降级。
6. `delivery`：Telegram、提醒和 AI Bot。

首阶段可继续使用 SQLite，但市场快照与事件表必须有索引、保留策略和迁移脚本。当资产量、写入频率或多实例部署超过单机能力时再迁移 PostgreSQL/TimescaleDB，不在 UI 阶段提前重构全部存储。

### 17.2 前端分层

- `app/*`：路由与页面编排。
- `features/radar`、`features/funds`、`features/info`、`features/agents`：业务模块。
- `components/ui`：Button、Tabs、Panel、Badge、Table、Drawer、State。
- `lib/api`：API 客户端、缓存、错误映射和请求取消。
- `lib/types`：由 API Schema 生成或集中维护的类型。
- `lib/realtime`：SSE、断线重连和可见性控制。
- `styles/tokens`：明暗主题和密度变量。

建议保留 Next.js 与 Tailwind；金融 K 线使用 Lightweight Charts 或合法的 TradingView 嵌入，复杂表格使用 TanStack Table，其他图形优先使用轻量 SVG，避免引入过重组件库。

## 22. 最终验收标准

达到以下条件才可称为“完整的 Mercu 同类型能力复刻”：

1. 雷达、资金、信息、AI、自选和单币详情六条核心路径全部可用。
2. 雷达首屏同时呈现实时事件、榜单和全场态势，而不是普通卡片列表。
3. 每条重点事件都具备事实、三类排名、共振、生命周期、来源和时间。
4. 现货、合约、OI、费率和价格口径明确，缺失时正确降级。
5. 从 Web 到 Telegram/AI Bot 的深链与提醒闭环可验证。
6. 390px 移动端无整体横向滚动，详情和筛选可单手操作。
7. 明暗主题、加载、空、错误、过期与断线状态全部有设计和测试。
8. AI 结论可追溯证据，不在事实缺失时给出确定性判断。
9. 自动化测试、部署验收、可观测性和回滚开关齐全。
10. 不使用 Mercu 的源码、私有接口、受限内容、品牌素材或未授权数据。

## 23. 建议的第一开发批次

第一批不立即做全部页面，先完成最能体现产品价值且能复用现有能力的垂直闭环：

2. 生成价格、OI、合约资金、现货资金四组正反榜单。
3. 将 `/radar` 重构为三栏驾驶舱。
4. 扩展现有详情抽屉和 Telegram 深链。
