当前必须修正的风险：

- `main()` 启动时会调用 `cleanup_obsolete_files()`，不应允许主入口自动删除任何文件。
- 默认运行模式是 `once`，会真实扫描，命中条件后可能发 Telegram。
- Telegram 发送函数直接拼 Bot API URL，业务逻辑直接调用发送函数。
- 外部请求分散在多个函数里，`requests.get` 和 `httpx.AsyncClient` 混用。
- 高风险接口 `openInterestHist` 在多个扫描逻辑中直接调用，缺少统一预算和熔断。
- 状态文件有些在根目录，有些在 `data/`，需要收敛。
- 单文件过大，继续堆功能会很难维护。

## 2. 新项目目标

项目定位：

轻量级加密市场观察雷达。

不做：

- Web/UI
- admin 查询
- 图表截图
- 自动交易
- 自动清理源码

核心目标：

- 发现公告机会和风险。
- 发现资金费率、持仓、价格、成交量异常。
- 发现可能启动前的暗流信号。
- 在真正启动时即时提醒。
- 定时推送高密度资金雷达摘要。
- 控制外部接口压力。
- 控制 Telegram 噪音。
- 默认 dry-run，真实发送必须确认。

## 3. 建议工程结构

不拆成很多文件，也不继续做 3000 行巨型单文件。

建议控制在 6 个 Python 文件：

```text
泡泡抓币/
  main.py          # CLI、运行模式、调度
  config.py        # 配置读取、默认值、敏感值隐藏
  storage.py       # JSON/SQLite 状态读写、原子写
  data_sources.py  # HTTP、Binance、CoinGecko、缓存、限流、熔断
  telegram.py      # dry-run、真实发送、分片、去重、限流、history
  radar.py         # 业务逻辑：公告、资金雷达、启动雷达、背离分析
  data/
  requirements.txt
  .env.oi
```

这种结构的目的：

- 文件少，仍然容易找。
- 每个文件职责明确。
- 不把请求、发送、状态写入和业务判断混在一起。
- 后续测试和调试成本低。

## 4. 最终保留的核心功能

### 4.1 公告和风险监控

用途：

- 监听 Binance 公告。
- 识别新币、Alpha、Launchpool、HODLer、Wallet、重要活动。
- 识别下架、移除交易对、停止交易等风险。

推送：

- 机会公告：有新事件时推。
- 风险公告：高优先级立即推。

周期：

- 每 60 秒扫描一次。

限流：

- 同公告不重复。
- 同币同类事件 24 小时冷却。

### 4.2 资金雷达摘要

用途：

定时生成类似“庄家雷达”的综合榜单。

包含模块：

- 负费率榜：找空头拥挤、潜在空头燃料。
- 综合榜：费率、市值、横盘、持仓综合评分。
- 埋伏池：低市值、横盘、持仓暗流、费率配合。
- 动量池：不要求横盘，找持仓、成交、价格正在动的币。
- 新币池：短历史币单独看，不和老币混排。
- 背离雷达：持仓变化和价格变化不同步的信号。
- 值得关注：从榜单重合、费率恶化、暗流、极端背离中提炼 3-5 条结论。
- 数据质量：扫描数量、持仓历史成功率、熔断状态、跳过数量。

周期：

- 内部数据刷新：10-15 分钟。
- Telegram 摘要推送：30-60 分钟，有变化才推。

限流：

- 每天最多 6 条摘要。
- 最短 30 分钟一次。
- 数据质量不足时只写状态，不推 Telegram。

### 4.3 启动雷达即时提醒

用途：

抓“币启动前”和“启动瞬间”。

分三段：

1. 提前预警：
   - 持仓增加。
   - 价格没明显动。
   - 成交额开始放大。
   - 资金费率没有过热。
   - 适合加入观察。

2. 启动确认：
   - 价格突破短周期区间。
   - 成交额继续放大。
   - 持仓没有明显回落。
   - 之前已经有提前预警。

3. 启动瞬间：
   - 5m/15m 快速拉升。
   - 成交量显著放大。
   - 持仓同步变化。
   - 突破最近高点。

周期：

- 启动瞬间扫描：1-3 分钟。
- 提前预警扫描：5-10 分钟。
- 启动确认扫描：5 分钟。

限流：

- 同币同阶段 6 小时冷却。
- 低置信只写状态。
- 从提前预警到启动确认发生状态变化时才推。

状态机：

```text
idle -> watching -> primed -> breakout -> launched
                         \-> failed
```

状态含义：

- `idle`：无信号。
- `watching`：提前预警。
- `primed`：多次暗流，接近启动。
- `breakout`：启动确认。
- `launched`：启动瞬间已触发。
- `failed`：启动失败。

### 4.4 背离分析

用途：

计算：

```text
背离度 = 持仓变化% - 价格变化%
```

识别：

- 持仓暴涨价格不动：疑似建仓。
- 持仓价格双升：趋势共振。
- 持仓上涨价格下跌：压制、洗盘或风险。
- 持仓下降价格上涨：空头止损推动。
- 持仓下降价格下跌：恐慌抛售。
- 极端背离：爆仓、控盘、剧烈波动风险。

周期：

- 1-2 小时。

推送方式：

- 默认放入资金雷达摘要。
- 极端背离可以独立即时推。

### 4.5 系统状态和异常

用途：

- 启动时确认 bot 已运行。
- 外部接口熔断时提醒。
- Telegram 发送失败时提醒。
- 数据质量过低时提醒。

周期：

- 启动时。
- 异常时。

限流：

- 同类异常 30 分钟冷却。

## 6. 外部接口和数据源

必须使用：

- Binance Futures `exchangeInfo`：合约列表、交易状态、上线时间辅助。
- Binance Futures `ticker/24hr`：价格、24h涨跌、24h成交额。
- Binance Futures `premiumIndex`：当前资金费率。
- Binance Futures `fundingRate` 或本地快照：费率趋势。
- Binance Futures `openInterestHist`：持仓变化。
- Binance Futures `klines`：价格变化、横盘天数、短周期突破、成交量放大。

可选使用：

- CoinGecko：市值、FDV、项目基础信息。
- Binance Square：热度辅助，不作为核心前置条件。

必须保护的接口：

- `openInterestHist`

保护策略：

- 每轮请求预算。
- 只对候选币请求。
- 连续 403/429 熔断。
- 失败降级，不阻断主扫描。
- 请求结果缓存。
